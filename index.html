04 Bridge · Now Identity

Identity Resolution

Data is clean. Now the platform decides which records belong to the same person — and which don't.

  • 4 match rules — fuzzy name + email, fuzzy name + phone, exact email, exact phone
  • Never name alone — Houssam and Hussam stay separate even with similar spelling
  • Reconciliation rules — per-field source authority. Opera wins for name, Loyalty wins for tier
  • Match clusters visible — every unification auditable in Profile Explorer
Outcome: 21 source records → 12 unified individuals. Houssam unified across 4 records. Hussam held separate. Rules sound.
MATCH RULE EXAMPLE
IF fuzzy(name) AND exact(email_normalised)
  THEN merge_records()

IF fuzzy(name) AND exact(phone_normalised)
  THEN merge_records()

NEVER match(name) alone

The unified profile becomes useful through metrics. Lifetime Value, stay frequency, F&B engagement, loyalty balance — every KPI in your RFP page 21.

  • 3 active CIs producing 14 governed metrics
  • SQL or Visual Builder — author by code or drag-drop
  • Schedule or on-demand — refresh strategies per insight
  • Auditable — every metric traceable back to source records
For Houssam: LTV AED 146,700 · 4 stays · 13 nights · ADR 11,285 — computed live.
CALCULATED INSIGHT SQL
SELECT
  guest_id__c,
  SUM(spend) AS lifetime_value,
  COUNT(stay_id) AS stays,
  SUM(nights) AS total_nights,
  MAX(check_in_date) AS last_stay
FROM Stay_DMO__dlm
GROUP BY guest_id__c
